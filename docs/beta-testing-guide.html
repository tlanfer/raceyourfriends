Race Your Friends / Beta Testing Guide

CI/CD setup for distributing builds to beta testers via TestFlight (iOS) and Google Play Internal Testing (Android). Everything runs through GitHub Actions — no local builds needed.

How It Works

Push a git tag to trigger the CI pipelines:

Tag PatterniOSAndroid
v1.0.0-beta.1 Upload to TestFlight → internal testers get it automatically Upload to Play Store internal testing track
v1.0.0-rc.1 Upload to TestFlight → you manually submit for external beta review Upload to Play Store internal testing track

Version numbers are derived from the tag automatically. v1.0.0-beta.3 sets the marketing version to 1.0.0 and the build number to the GitHub Actions run number (always incrementing).

Part 1: iOS / TestFlight

1 Create Apple Developer Account

Enroll in the Apple Developer Program ($99/year). This is required for code signing and TestFlight.

Enrollment can take up to 48 hours. Once approved, you'll have access to App Store Connect and the developer portal.

2 Create the App in App Store Connect

Go to App Store Connect → My Apps → + → New App:

FieldValue
PlatformiOS
NameRace Your Friends
Primary LanguageEnglish (U.S.)
Bundle IDRegister new → com.raceyourfriends.app (Explicit)
SKUraceyourfriends
Bundle ID must be exactly com.raceyourfriends.app — this matches the appId in capacitor.config.ts. If you use a different Bundle ID, the build will fail signing.

When registering the Bundle ID, enable the Location Services capability (required for GPS tracking).

3 Generate an App Store Connect API Key

The CI pipeline uses an API key (instead of certificates/profiles) for automatic code signing and upload. This is the simplest approach — Xcode manages certificates and provisioning profiles automatically via the API key.

  1. Go to App Store Connect → Users and Access → Integrations → App Store Connect API
  2. Click Generate API Key (or +)
  3. Name: GitHub Actions
  4. Access: App Manager (needs permission to manage signing and upload builds)
  5. Click Generate

You'll get three values. Save all of them — the .p8 file can only be downloaded once:

ValueWhere to find itExample
Issuer IDTop of the API Keys page69a6de7e-....
Key IDIn the key row2X9R4HXF34
API Key fileDownload the .p8 fileAuthKey_2X9R4HXF34.p8

Base64-encode the .p8 file for storage as a GitHub secret:

# macOS/Linux
base64 -i AuthKey_XXXXXXXX.p8

# Windows (PowerShell)
[Convert]::ToBase64String([IO.File]::ReadAllBytes("AuthKey_XXXXXXXX.p8"))

4 Add GitHub Secrets

Go to your GitHub repo → Settings → Secrets and variables → Actions. Add these repository secrets:

Secret NameValue
APP_STORE_CONNECT_KEY_IDThe Key ID from step 3 (e.g. 2X9R4HXF34)
APP_STORE_CONNECT_ISSUER_IDThe Issuer ID from step 3
APP_STORE_CONNECT_API_KEY_BASE64Base64-encoded contents of the .p8 file
VITE_WS_URLYour production WebSocket URL (e.g. wss://your-server.example.com/ws) — likely already set
That's all the iOS CI needs. The API key lets Xcode automatically create and manage distribution certificates and provisioning profiles on the CI runner. No manual certificate or profile management required.

5 Set Up TestFlight Testers

Internal Testing (beta tags)

Internal testers are members of your App Store Connect team. They get every build automatically — no review needed.

  1. In App Store Connect → Users and Access
  2. Add team members by email (they must have Apple IDs)
  3. Give them any role (e.g., Marketing, Developer)
  4. Go to your app → TestFlight → Internal Group
  5. Create a group (e.g., "Core Team") and add your team members
  6. They'll get an email to install TestFlight and the app

Up to 100 internal testers. Builds appear in TestFlight within 5-30 minutes of upload.

External Testing (RC tags)

For wider distribution (up to 10,000 testers), you need an external testing group. Builds require Beta App Review (usually 24-48 hours the first time, faster after that).

  1. Go to your app → TestFlight → click + next to External Groups
  2. Name the group (e.g., "Beta Testers")
  3. Add testers by email or enable a Public Link

When a build from an RC tag is uploaded, the CI job summary will remind you to submit it for external review. In App Store Connect:

  1. Go to your app → TestFlight
  2. Wait for the build to finish processing
  3. Select the build under your external group
  4. Fill in the "What to Test" field and compliance information
  5. Click Submit for Review

Part 2: Android / Google Play

1 Create Google Play Developer Account

Register at Google Play Console ($25 one-time fee). Identity verification may take several days for new accounts.

2 Generate a Release Keystore

Run this once on any machine. The keystore signs your app — you'll need the same keystore for every future release.

keytool -genkeypair -v \
  -storetype PKCS12 \
  -keystore raceyourfriends-release.keystore \
  -alias raceyourfriends \
  -keyalg RSA \
  -keysize 2048 \
  -validity 10000

You'll be prompted for a password and identity details. Remember the password and alias.

Then base64-encode it for GitHub:

# macOS/Linux
base64 -i raceyourfriends-release.keystore

# Windows (PowerShell)
[Convert]::ToBase64String([IO.File]::ReadAllBytes("raceyourfriends-release.keystore"))
Store the keystore file and password securely (e.g., password manager). If you lose the keystore, you cannot push updates to the same app listing. Do NOT commit it to git.

3 Create the App in Google Play Console

  1. Go to Google Play Console → Create app
FieldValue
App nameRace Your Friends
Default languageEnglish (United States)
App or gameGame
Free or paidFree

Accept the declarations and create the app. You'll need to fill in the content rating questionnaire and a few other setup steps before you can publish to any track — Google Play Console will guide you through what's missing.

Opt in to Google Play App Signing when prompted. This lets Google manage your distribution key while you keep an upload key. It's the recommended setup and is required for new apps.

4 Create a Google Play Service Account

The CI pipeline needs API access to upload builds to Google Play. This requires a Google Cloud service account linked to Play Console.

  1. Go to Google Cloud Console
  2. Create a project (or use an existing one)
  3. Enable the Google Play Android Developer API
  4. Go to IAM & Admin → Service Accounts → Create Service Account
  5. Name it (e.g., github-actions-play-upload), no roles needed in GCP
  6. Click the service account → Keys → Add Key → Create new key → JSON
  7. Download the JSON key file — this is your GOOGLE_PLAY_SERVICE_ACCOUNT_JSON secret

Then link the service account to Google Play Console:

  1. Go to Google Play Console → Settings → API access
  2. Link your Google Cloud project
  3. Find your service account and click Manage Play Console permissions
  4. Grant Release to testing tracks permission for your app
  5. Click Invite user → Send invite

5 Add GitHub Secrets

Go to your GitHub repo → Settings → Secrets and variables → Actions. Add these repository secrets:

Secret NameValue
ANDROID_KEYSTORE_BASE64Base64-encoded keystore file from step 2
ANDROID_KEYSTORE_PASSWORDThe password you set when generating the keystore
ANDROID_KEY_ALIASraceyourfriends (if you used the command above)
ANDROID_KEY_PASSWORDSame as keystore password (unless you set them differently)
GOOGLE_PLAY_SERVICE_ACCOUNT_JSONFull contents of the JSON key file from step 4
VITE_WS_URLYour production WebSocket URL — likely already set

6 Set Up Internal Testers

  1. In Google Play Console → your app → Testing → Internal testing
  2. Go to the Testers tab
  3. Create an email list (e.g., "Beta Testers")
  4. Add tester emails (must be Google accounts)
  5. Save, then copy the opt-in link from the top of the page
  6. Share the opt-in link with your testers

Testers click the opt-in link, accept, then install from the Play Store link provided. Up to 100 testers per list. Builds are available within minutes of upload.

First upload: You need to manually upload the very first AAB through the Play Console UI before the API can upload subsequent builds. Run the CI once, download the AAB from the GitHub Actions artifacts, and upload it manually. After that, CI uploads will work automatically.

Triggering Builds

Once setup is complete, deploy by pushing a tag:

# Beta build (internal testers only)
git tag v1.0.0-beta.1
git push origin v1.0.0-beta.1

# Release candidate (submit for external testing after upload)
git tag v1.0.0-rc.1
git push origin v1.0.0-rc.1

Both iOS and Android pipelines run in parallel. Check the Actions tab for progress. Each run produces:

  • iOS: IPA uploaded to TestFlight + artifact saved in GitHub Actions
  • Android: AAB uploaded to Play Store internal testing + artifact saved in GitHub Actions
  • Job summary with version info and any manual steps needed

Project Files Changed for CI

These files were modified to support CI/CD beta distribution:

.github/workflows/ios.yml

Replaced the unsigned simulator build with a full archive + TestFlight upload pipeline. Uses the App Store Connect API key for automatic code signing (no manual certificate/profile management). Triggers on v*-beta* and v*-rc* tags. Extracts version from the tag and sets build number to the GitHub Actions run number.

.github/workflows/android.yml

Replaced the debug APK build with a signed release AAB build + Google Play internal testing upload. Decodes the keystore from a base64 secret, builds with bundleRelease, and uploads via the r0adkll/upload-google-play action. Same tag triggers as iOS.

android/app/build.gradle

Added conditional release signing configuration that reads from environment variables (ANDROID_KEYSTORE_PATH, ANDROID_KEYSTORE_PASSWORD, ANDROID_KEY_ALIAS, ANDROID_KEY_PASSWORD). When these aren't set (local dev), the release build type falls back to unsigned. Also added CI-injectable version: ANDROID_VERSION_CODE and ANDROID_VERSION_NAME env vars override the defaults.
